🔒 P0 Security Fix: Tool Profiles Now Actually Work

We discovered that Claude Bridge's readonly tool profile was completely ineffective. The --allowedTools CLI flag, which we used to restrict Claude's tool access, does nothing in -p (non-interactive) mode. Claude had full access to Bash, Write, and Edit regardless of the selected profile.

Root cause: --allowedTools is a permission whitelist that pre-approves tools for interactive confirmation. But -p mode skips all interactive confirmation, making the flag meaningless.

Fix: Switched to --tools, which controls the actual available tool set. Tools excluded by --tools completely disappear from Claude's perspective — it can't see or use them.

ProfileTools AvailableUse Case
readonly (default)Read, Grep, Glob, WebSearch, WebFetchSafe queries, no side effects
standardAll tools (default)Full access for trusted operations
restrictedRead, Grep, GlobStrictest — no Bash, no network

Important finding: --tools doesn't support fine-grained Bash patterns like Bash(ls:*). It's all-or-nothing — either Bash is available or it isn't. This is a Claude Code CLI limitation.

⚡ New Feature: /task — Two-Phase Task Orchestration

The new /task command implements a "Plan D" architecture: readonly analysis → user confirmation → full execution.

How it works

/task Check disk usage and clean up files larger than 100MB in /tmp
  1. Phase 1 (readonly): Claude analyzes the situation using only read tools. It reads files, checks status, and produces a plan — but cannot execute any changes.
  2. Phase 2 (confirm): The analysis is sent to Telegram with [Execute] / [Cancel] buttons. You review the plan.
  3. Phase 3 (execute): If you click Execute, Claude resumes the same session with full tool access and carries out the plan.

Why this design?

We evaluated four architectures for multi-step task orchestration:

  • Plan A: Let Claude handle everything in one shot → no safety gate for destructive operations
  • Plan B: CB parses Markdown into steps → brittle, format-dependent
  • Plan C: Two-phase with Claude parsing steps into JSON → orchestrator-on-orchestrator (Claude Code already has an agentic loop)
  • Plan D (chosen): Readonly analysis + confirmation + full execution → simplest, ~75 lines of code, leverages --resume for state continuity

Key insight: in readonly mode, Claude gracefully degrades to text suggestions. When it can't execute rm or git push, it doesn't crash — it simply tells you what it would do and recommends you run the command yourself. This behavior is exactly what we need for the analysis phase.

Technical Details

  • /task supports both inline text (/task do X) and reply-to-message (/task as a reply)
  • Uses the same session (--resume) across phases, so Claude retains full context
  • No new database tables — session state is sufficient
  • Cost tracking works for both phases independently
